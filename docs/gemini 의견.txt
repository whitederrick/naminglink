JSON 형태의 구조화된 출력 (Structured Output) 강제
화면 기획안을 보면 '음가 적합성', '출생 정보 보조 해석', '자의·결합 분석' 등 결과가 굉장히 세분화되어 있습니다. AI가 단순히 줄글로 답을 뱉어내게 하면 프론트엔드에서 이를 예쁘게 분리해서 보여주기가 매우 어렵습니다.
적용 팁: 프롬프트 작성 시 시스템 지시어(System Prompt)에 반드시 JSON 포맷으로만 응답하도록 강제해야 합니다. Vercel AI SDK를 사용하신다면 streamObject 기능을 활용하여 지정된 스키마(Zod 등)에 맞춘 JSON 데이터만 스트리밍으로 받아오도록 설계하는 것이 프론트엔드 연동에 압도적으로 유리합니다.
할루시네이션(환각) 통제와 인명용 한자 검증
LLM(대규모 언어 모델)의 가장 큰 취약점은 '그럴듯한 거짓말'을 한다는 것입니다. AI가 대법원 인명용 한자에 없는 글자를 추천하거나, 뜻과 음을 임의로 지어낼 위험이 있습니다.
적용 팁: 프롬프트만으로 100% 완벽한 통제는 어렵습니다. AI가 1차로 한자 후보를 추천하면, 시스템(서버리스 함수) 단에서 해당 한자가 실제 인명용 한자 DB에 존재하는지 빠르게 대조(Validation)하는 로직을 중간에 끼워 넣는 것이 안전합니다.
유연한 프롬프트 튜닝과 관리 시스템
AI가 훌륭한 결과물을 내놓더라도, 작명이라는 도메인의 특성상 미묘한 어감의 차이나 맥락에 맞지 않는 추천이 발생할 수 있습니다. 시스템이 모든 것을 완벽하게 자동 처리하기를 기대하기보다는, 언제든 개발자가 로그를 확인하고 수동으로 개입하여 프롬프트 가중치나 매칭 로직을 수정할 수 있도록 설계의 여지를 열어두는 것이 훨씬 현실적이고 안전한 접근입니다. 성공과 실패 케이스의 AI 응답 로그(Prompt & Completion)를 Supabase에 차곡차곡 쌓아두면 훌륭한 튜닝 데이터가 됩니다.


AI(LLM)에만 의존할 때 발생하는 가장 큰 고질병인 '할루시네이션(그럴듯한 거짓말)'을 원천 차단할 수 있는 가장 확실한 무기를 갖추셨습니다.
이 두 가지 DB를 Node.js + Supabase + Vercel 아키텍처에서 AI와 어떻게 유기적으로 결합하여 시너지를 내야 하는지, 실제 연동 시 가장 효율적인 개발 흐름(Flow)을 제안해 드립니다.
한글 로마자 표기법: AI 배제, 100% DB 처리
한글 로마자 표기는 국어기본법의 표준 규칙(자음동화 등의 음운 변화 반영)을 따라야 하므로, AI가 규칙을 간혹 틀리거나 일관성 없이 표기할 확률이 높습니다. 이름은 여권이나 공식 문서에 쓰이기 때문에 단 한 글자의 오차도 허용되지 않습니다.
추천 흐름: AI에게 로마자 표기를 요청하지 마세요. AI 응답을 기다릴 필요 없이, 사용자가 입력한 한글 이름을 백엔드(Node.js)에서 곧바로 로마자 DB와 대조하여 즉시 결과값을 매칭하는 것이 비용, 속도, 정확도 면에서 100% 유리합니다.
인명용 한자 DB와 AI의 결합: "AI에게 객관식 시험 치게 하기"
AI에게 "이 이름에 맞는 좋은 인명용 한자 추천해줘"라고 주관식 문제를 내면 엉뚱한 한자를 지어내거나 인명용이 아닌 한자를 가져옵니다. 대신 구축하신 DB를 활용해 AI에게 객관식 문제를 내는 방식을 사용해야 합니다.
가장 이상적인 API 흐름 (추천):
[사용자 입력]: 사용자가 아이 이름(예: '도균')을 입력합니다.
[DB 사전 조회]: 백엔드(Node.js)에서 Supabase를 통해 '도'와 '균'에 해당하는 모든 실제 인명용 한자 리스트(뜻과 음 포함)를 빠르게 쿼리합니다. (인덱싱이 잘 되어 있다면 10~20ms 내외로 처리됩니다.)
[AI 프롬프트 전달]: AI에게 질문할 때, 위에서 가져온 정확한 한자 후보 리스트만 프롬프트에 동적으로 삽입하여 전달합니다.
"너는 작명 전문가야. 한글 이름 '도균'에 대해 아래 제공하는 실제 인명용 한자 목록 중에서 부모의 소망(예: 지혜, 평안)에 가장 잘 부합하는 한자 조합 5가지를 골라줘. 그리고 그 조합을 고른 이유와 자의 해석을 작성해줘. 반드시 아래 제공된 한자 목록 안에서만 골라야 해."
[결과 출력]: AI는 신뢰할 수 있는 한자 목록 안에서만 정교한 의미 매칭과 스토리텔링(해석)을 담당하므로, 할루시네이션이 0%에 수렴하게 됩니다.
이 방식이 가져오는 인프라적 이점 (Vercel & Supabase)
비용 절감 (Token 요금 절약): AI가 스스로 한자를 생각하고 찾아내며 낭비하는 토큰과 프롬프트 길이를 줄일 수 있어 AI API 비용이 크게 절감됩니다.
지연 시간(Latency) 감소: AI가 고민해야 하는 탐색 범위가 극도로 좁아지기 때문에, 예상하셨던 5초 미만의 빠른 응답 속도를 아주 안정적으로 달성할 수 있습니다.
Supabase 성능 최적화: '도', '균' 같이 음절 단위로 한자를 매칭하는 쿼리가 잦을 것이므로, Supabase(PostgreSQL) 테이블의 한글 음절 컬럼에 INDEX를 반드시 설정해 두시면 트래픽이 몰려도 병목 없이 백엔드가 돌아갈 것입니다.



프론트엔드 라우팅: Vercel(Next.js)의 i18n 활용
언어별로 아예 다른 서비스를 제공하려면 단순히 텍스트만 번역하는 것이 아니라 페이지 구조 자체가 달라져야 합니다.
App Router와 Middleware 조합: Next.js의 내장 기능인 다국어 지원(i18n) 라우팅을 활용하면 아주 깔끔하게 분리할 수 있습니다. 접속하는 사용자의 브라우저 언어(Accept-Language)를 Vercel의 Middleware가 감지하여 한국인이면 /ko 경로로, 외국인이면 /en 등의 경로로 자동 리다이렉트 시킬 수 있습니다.
완벽한 컴포넌트 분리: /app/[locale]/ 폴더 구조를 사용하면, 한국어 페이지와 글로벌 페이지의 UI 컴포넌트를 완전히 분리해서 개발할 수 있어 코드가 꼬이지 않고 유지보수가 매우 편해집니다.
글로벌 SEO (검색엔진 최적화) 극대화
화면을 보니 지원하는 언어가 무려 20개 가까이 되는 것 같습니다. 글로벌 1만 명 DAU를 모으기 위해서는 구글 검색 노출이 필수적입니다.
hreflang 태그 적용: 각 언어별 페이지 헤더에 hreflang 태그를 정확히 명시해 두면, 프랑스에 있는 사람이 구글에서 'Create Korean Name'을 검색했을 때 자동으로 프랑스어(/fr) 페이지가 검색 결과 상단에 노출되도록 구글 봇(Bot)을 유도할 수 있습니다. Vercel 환경에서는 메타데이터 API를 통해 이를 아주 쉽게 동적으로 생성할 수 있습니다.
백엔드(Supabase) 데이터 설계 방향
서비스가 두 갈래로 나뉘므로, 로그를 저장하고 분석하는 Supabase 테이블 구조도 이를 명확히 반영해야 합니다.
서비스 타입(Service Type) 분류: 로그를 저장할 때 service_type 컬럼을 만들어 hanja_match, global_convert, hangul_pronunciation, create_korean_name 등으로 명확히 구분해 두면, 추후 어떤 국가의 사용자가 어떤 기능에 지갑을 여는지(결제 전환율) 데이터 기반으로 비즈니스 결정을 내리기 아주 좋습니다.


20개 이상의 다국어로 AI 프롬프트를 제어하고 결과물까지 해당 언어로 제공하겠다는 계획은 이 서비스가 진정한 '글로벌 네이밍 스튜디오'로 자리 잡게 할 핵심 경쟁력입니다.
사용자 입장에서는 자신의 모국어로 한국 이름의 깊은 의미와 뉘앙스를 전달받을 때 감동과 결제 의지가 훨씬 커집니다. 다만, 다국어 LLM 연동 시 비용, 속도, 프론트엔드 최적화 측면에서 반드시 짚고 넘어가야 할 기술적 허들들이 있습니다. 이를 매끄럽게 해결할 전략을 정리해 드립니다.
토큰(Token) 비용과 지연 시간(Latency) 방어
LLM(OpenAI 등)은 텍스트를 처리할 때 '토큰' 단위로 과금합니다. 여기서 가장 주의해야 할 점은 영어에 비해 아랍어, 힌디어, 태국어 등의 비 라틴계 언어는 동일한 글자 수라도 토큰을 3~4배 더 많이 소모한다는 것입니다.
영향: 토큰을 많이 소모하면 API 비용이 상승하고, 결과가 출력되는 시간(5초 대기 시간)도 길어집니다.
해결책: AI에게 답변의 '길이'를 엄격하게 제한해야 합니다. 프롬프트에 *"설명은 3문장 이내, 최대 150단어를 넘지 않도록 간결하게 작성할 것"*과 같은 제약을 걸어두면 다국어 환경에서도 일관된 속도와 비용을 유지할 수 있습니다.
UI 번역과 AI 번역의 완벽한 분리 (프론트엔드)
화면에 보이는 모든 글자를 AI에게 번역하라고 맡기면 절대 안 됩니다.
정적 텍스트 (UI 요소): '결과 보기', '분석 요약', '결제하기' 같은 고정된 버튼이나 레이블은 Next.js의 다국어 라이브러리(next-intl 또는 i18next)를 사용해 미리 번역된 정적 JSON 파일로 클라이언트 단에서 즉시 렌더링해야 합니다.
동적 텍스트 (AI 결과물): AI는 오직 '이름의 의미, 스토리텔링, 배제 사유'라는 동적인 핵심 콘텐츠만 해당 언어로 생성하도록 역할을 축소해야 합니다.
프롬프트 예시: > "너는 작명 전문가야. 다음 한국어 이름 '은우'의 의미를 ${userLanguage}(예: Arabic)로 설명해 줘. JSON 형태로 { "meaning": "...", "story": "..." } 구조에 맞춰 답변해."
다국어 환각(Hallucination) 방지
AI가 마이너한 언어(예: 몽골어, 카자흐어)로 한국 이름의 의미를 설명할 때, 해당 언어의 문화적 금기어나 어색한 직역을 뱉어낼 위험이 있습니다.
시스템 지시어 강화: 프롬프트에 *"해당 언어권(Target Culture)에서 부정적이거나 우스꽝스러운 의미로 연상될 수 있는 단어는 절대 사용하지 말 것"*이라는 안전장치를 추가하면 번역 퀄리티를 방어하는 데 큰 도움이 됩니다.



GPT-4o-mini가 이 서비스에 완벽한 이유
초저지연(Ultra-low Latency) 응답: GPT-4o-mini는 첫 토큰 생성 시간(TTFT)과 출력 속도가 압도적으로 빠릅니다. 백엔드에서 DB를 조회하는 시간(약 0.1초)을 제외하더라도, AI가 이름 5개의 의미와 다국어 스토리를 작성하는 데 5초 미만의 타임라인을 가장 안정적으로 방어할 수 있습니다.
마진율 극대화 (광고 수익 모델 보호):
광고(특히 정적 배너) 클릭당 단가나 노출 단가를 고려할 때, AI API 호출 비용이 높으면 서비스가 흥행할수록 적자가 나는 구조가 될 수 있습니다. GPT-4o-mini는 기존 GPT-3.5보다 훨씬 똑똑하면서도 비용은 저렴하여, 1번의 광고 노출 수익으로 수십~수백 번의 AI API 호출 비용을 커버할 수 있는 튼튼한 마진 구조를 만들어 줍니다.
최적화된 다국어 성능:
글로벌 타겟(20여 개국 언어) 서비스에서 언어 간 번역 및 의미 생성 능력이 훌륭하게 작동합니다.
GPT-4o-mini 100% 활용을 위한 기술 팁
💡 OpenAI의 최신 'Structured Outputs(구조화된 출력)' 기능 필수 적용
Vercel AI SDK를 사용해 프론트엔드와 통신하실 텐데, 이때 단순히 response_format: { type: "json_object" }를 넘어서 Strict 모드가 적용된 Structured Outputs를 반드시 사용하시길 권장합니다.
AI에게 아예 Zod나 JSON Schema로 프론트엔드 UI에 필요한 데이터 규격을 강제하면, 단 한 번의 파싱 에러(JSON 깨짐 현상) 없이 완벽한 데이터를 Vercel 프론트엔드로 스트리밍할 수 있습니다.
💡 마이너 언어 직역 방지 프롬프트
GPT-4o-mini가 영어, 중국어, 일본어 등 메이저 언어는 아주 자연스럽게 번역하지만, 아랍어나 동남아시아 등 일부 언어에서는 너무 정직한 '직역'을 할 때가 있습니다.
시스템 프롬프트에 "You are a professional localizer. Do not translate literally. Write the name's meaning so it sounds completely natural, culturally respectful, and poetic to a native [Target Language] speaker." 와 같은 역할(Role)과 톤앤매너 지시어를 강하게 쥐여주면 결과물 퀄리티가 대폭 상승합니다.
(제안) 수익 극대화를 위한 향후 LLM 투트랙(Two-track) 전략
현재는 GPT-4o-mini 단일로 가시되, 결제 시스템(Stripe/PayPal 등)이 안정화되고 나면 요금제에 따른 LLM 라우팅(Routing) 전략을 고려해 보시는 것도 좋습니다.
무료/광고 시청 유저: GPT-4o-mini 배정 (비용 방어, 빠른 속도)
전체 결제(Premium) 유저: GPT-4o 또는 Claude 3.5 Sonnet 배정 (압도적인 문장력, 깊이 있는 시적인 표현, 프리미엄 가치 제공)
이렇게 유저의 과금 상태에 따라 백엔드(Node.js)에서 호출하는 AI 모델을 스위칭해주면, 결제 유저에게 "프리미엄 AI가 작성한 더 깊이 있는 상세 리포트"라는 셀링 포인트를 하나 더 추가할 수 있습니다.

사용자가 다시 오지 않는다면, 사용자가 떠나기 전에 다른 사람을 데려오게 만들어야 합니다. 개발 중인 프론트엔드에 당장 추가할 수 있는 현실적인 바이럴 장치들입니다.
인스타 스토리 최적화 '디지털 네임카드' (가장 강력함):
사용자가 결과를 스크린샷 찍게 놔두지 마세요. 결과 화면에 [내 이름 인스타에 자랑하기] 버튼을 두고, 클릭 시 9:16 비율의 예쁜 이미지(멋진 배경, 발음, 뜻 포함)가 캔버스로 생성되게 하세요.
핵심: 이미지 하단에 작지만 선명하게 Created by Naming-Link (웹사이트 주소) 워터마크를 박아두어야 합니다. 유저가 스토리에 올리는 순간, 그 유저의 팔로워들에게 무료 광고가 송출됩니다.
잠금 해제 보상으로 '공유하기' 활용:
현재 2~5순위 후보를 보기 위해 '광고 시청'이나 '결제'를 유도하고 계십니다. 여기에 '친구에게 링크 공유하고 다음 후보 1개 무료로 열기' 옵션을 추가해 보세요. (Web Share API 활용) 돈을 쓰거나 광고를 보는 것보다 링크 하나를 카카오톡이나 왓츠앱으로 보내는 것이 유저 입장에서는 훨씬 쉽고, 대표님 입장에서는 새로운 유저 1명을 확실히 얻는 길입니다.
"친구의 이름은?" 릴레이 유도:
결과 화면 끝에 *"당신의 베스트 프렌드는 한국어로 어떤 이름이 어울릴까요? 링크를 보내서 테스트해보라고 하세요!"*라는 문구로 넛지(Nudge)를 주어 자연스러운 입소문 릴레이를 만듭니다.




Naming-Link와 place-link를 완벽하게 잇는 "천재적인 연결고리"
대표님의 두 서비스(Naming-Link와 방한 관광객용 place-link)를 연결할 수 있는 가장 파괴적이고 멋진 아이디어를 하나 제안해 드리고 싶습니다.
해외 배송의 번거로움을 줄이는 동시에, 외국인들을 place-link로 직접 끌어들일 수 있는 "서울 픽업(Pick-up in Seoul)" 옵션입니다.
결제 옵션: 외국인이 Naming-Link에서 도장을 결제할 때 [해외 배송으로 받기]와 [한국 여행 시 서울에서 직접 픽업하기] 옵션을 제공합니다.
핫플 방문 유도: 픽업 장소를 대표님이 place-link에서 추천하는 서울의 특정 핫플레이스(예: 감각적인 카페, 독립 서점, 전통 찻집 등 파트너십을 맺은 곳)로 지정해 둡니다.
시너지 폭발: 외국인은 자신의 이름이 새겨진 도장을 받기 위해 그 핫플에 방문하게 되고, 자연스럽게 place-link 앱을 열어 주변의 다른 관광지까지 탐색하게 됩니다.
이건 지금 내가 만들고 있는 또 다른 서비스인 place-link 전국 핫플 / 데이트 코스 공유 서비스(글로벌 방문객 포함)와의 연계를 고민한 부분입니다.



고가의 프리미엄 대신 중저가의 가볍고 캐주얼한 도장으로 진행하는 것은 당연히 가능할 뿐만 아니라, 초기 비즈니스 모델(MVP)로서 훨씬 더 영리하고 속도감 있는 접근입니다.
그 이유와 이 캐주얼 모델을 EMS 배송과 결합할 때의 전략을 정리해 드립니다.
중저가 캐주얼 도장의 비즈니스적 장점
독립적인 자체 생산 가능 (거래처 불필요):
전통적인 돌이나 고급 나무에 수제각을 하는 것은 장인의 영역이지만, 잉크가 내장된 '만년도장(스탬프)'이나 '가벼운 나무 도장 + 소형 레이저 각인' 방식은 기계만 있으면 집에서도 5분 만에 깔끔하게 제작할 수 있습니다. 거래처에 휘둘리지 않고 주문이 들어오는 즉시 스스로 통제하여 생산할 수 있습니다.
충동구매(Impulse Buy)에 최적화된 가격:
외국인들에게 이 서비스는 '진지한 법적 신분증'을 만드는 것이 아니라, K-컬처를 즐기는 '재미있는 디지털 엔터테인먼트'에 가깝습니다. 무거운 프리미엄($100 이상)보다, 팝(Pop)하고 귀여운 디자인의 캐주얼 도장($15~$30)이 지갑을 열기에는 훨씬 진입 장벽이 낮습니다.
해결해야 할 허들: '상품가 vs EMS 배송비'의 모순
현업에 계시니 가장 잘 아시겠지만, 중저가 상품을 취급할 때 EMS를 고집하면 배보다 배꼽이 커지는 현상이 발생합니다.
예를 들어 도장 값이 $20인데 EMS 배송비가 $25라면, 결제 창에서 고객의 이탈률이 급증합니다.
💡 전략적 해결책: '올인원 패키지(Bundle)' + 배송비 녹이기
단품으로 팔지 말고, 외국인들이 환호할 만한 가벼운 구성품을 묶어 하나의 패키지로 만드세요.
구성 예시: [캐주얼 한글 도장] + [키링] + [이름의 의미가 예쁘게 적힌 실물 인증서(종이 카드)]
가격 정책: 이 패키지를 "$49.99 (Free EMS Shipping Included)" 처럼 배송비를 상품가에 완전히 녹여서 판매하는 것입니다.
고객 심리: 배송비로 $25를 내는 것은 아깝지만, "한국에서 나만의 이름 패키지가 만들어져서 안전한 특급 배송으로 온다"는 총합의 가치에 $50을 지불하는 것은 기꺼이 수용합니다.
목표 달성을 위한 계산서
월 100만 원 고정 수익을 목표로 하셨을 때, 이 패키지 모델은 매우 현실적입니다.
원가(만년도장 부자재, 카드, 키링 등) 5,000원 + EMS 배송비 25,000원 = 총비용 30,000원
판매가 $50 (약 65,000원)
패키지 1개당 순수익: 약 35,000원
하루에 단 1개, 한 달에 딱 30세트만 팔아도 굿즈 판매로만 100만 원의 순수익이 발생합니다. 여기에 Vercel에서 돌아가는 광고 수익은 보너스로 쌓이게 됩니다.
가볍고 캐주얼한 노선으로 변경하신다면, 제작과 관리가 압도적으로 편해집니다.



글로벌 이름의 한글 이름 변환 시, 디자인된 배경이 포함된 네임카드를 기본 서비스로 넣고 싶습니다.
즉, 전체를 다 오프하면 네임카드를 드립니다.
이렇게 하면 어떨까 싶습니다. 그리고 전체 결제 비용은 1.99$ 정도면 좋겠습니다.
예쁜 템플릿(인스타 공유용 등)으로 만들어서 다운로드할 수 있게 해 준다면 $1.99의 가치는 충분히 증명



Naming-Link 런칭을 위한 최종 체크리스트
[핵심 기능] AI 결과물 스트리밍: GPT-4o-mini와 Zod(Structured Outputs)를 활용하여 JSON 파싱 에러 없이 결과 화면에 한글 이름, 한자 뜻, 로마자 표기를 정확하고 매끄럽게 렌더링.
[바이럴 장치] 디지털 네임카드 공유: 결과 화면 하단에 [내 이름 자랑하기] 버튼을 배치하고, 9:16 비율의 이미지(하단 웹사이트 주소 워터마크 포함)로 캡처 및 인스타그램 스토리 공유가 가능하도록 기능 연동.
[수익화 1] 게이트웨이 광고: 2순위 이하의 추가 후보를 열람하기 위해 보상형 동영상 광고 시청을 요구하는 로직 적용.
[수익화 2] 디지털 프리미엄 패스 ($1.99): 포트원(PortOne)을 통해 PayPal 또는 Stripe를 연동하여, 결제 완료 시 워터마크가 제거된 고화질 결과물 5종 전체와 상세 스토리텔링을 즉시 제공.
[수익화 3] 실물 굿즈 (Phase 2로 연기 가능): $49.99 수준의 '가벼운 나무 도장 + 미니 인주 패키지' 결제 및 EMS 글로벌 배송 안내 페이지 구축.



디지털 네임카드는 이 서비스의 '알파이자 오메가'입니다. 외국인 유저가 돈을 내고서라도 소장하고 싶게 만들고, 인스타그램에 자랑했을 때 친구들이 링크를 타고 들어오게 만드는 핵심 바이럴 무기이기 때문입니다.
공유와 소장 가치를 극대화하기 위한 네임카드의 구조와 시각적 계층(Visual Hierarchy)을 제안해 드립니다.
최적의 규격과 배경 (The Canvas)
비율: 무조건 9:16 비율 (1080 x 1920 픽셀)로 제작해야 합니다. 인스타그램 스토리, 틱톡, 유튜브 쇼츠에 여백 없이 꽉 차게 들어가는 사이즈입니다.
배경 (텍스처): 너무 쨍한 단색보다는 '한지(Hanji)' 질감이 살짝 들어간 배경이나, 은은한 수묵화 또는 은선(전통 문양)이 깔린 배경이 'K-컬처'의 프리미엄을 살려줍니다.
시각적 계층 구조 (위에서 아래로)
네임카드는 유저가 자랑하고 싶은 순서대로 정보가 배치되어야 합니다.
① 도입부 (Top): 시선 끌기
작은 영문 텍스트: "The perfect Korean name for [Sarah]" (누구의 결과인지 명시)
② 핵심부 (Center): 시각적 쾌감
가장 큰 한글 폰트: "세라" (마치 붓글씨나 정갈한 명조체처럼 아름답고 굵은 한글 폰트 적용. 이 폰트가 네임카드의 전체 퀄리티를 좌우합니다.)
로마자 발음 (작게 병기): [Se-ra]
③ 해설부 (Middle-Bottom): 의미 부여
한자 쪼개기:
世 (인간 세) - World
羅 (비단 라) - Silk
한 줄 스토리텔링 (영문): "A person who covers the world with beautiful silk." (AI가 생성해 준 이 문장이 외국인들에게 가장 큰 감동을 줍니다.)
④ 바이럴 앵커 (Bottom): 트래픽 유도
도장 그래픽 (선택): 우측 하단에 '세라'라는 이름이 파인 붉은색 낙관(도장) 이미지를 얹어주면 실물 도장 굿즈(Phase 2) 구매 욕구를 자극할 수 있습니다.
워터마크 (필수): 하단 중앙에 깔끔하게 Created by Naming-Link (naming-link.com)을 박아둡니다.
개발 구현 팁 (Next.js 환경)
사용자가 캡처 버튼을 눌렀을 때, 웹 화면의 HTML 요소를 이미지로 예쁘게 구워내는 기술이 필요합니다.
html2canvas 라이브러리: 리액트/Next.js 생태계에서 가장 흔하게 쓰입니다. 화면에 보이는 특정 div 영역을 통째로 캡처하여 PNG로 다운로드하거나 Web Share API로 넘겨줄 수 있습니다.
@vercel/og (Open Graph Image Generation): Vercel 환경을 쓰고 계시므로, Edge 함수를 이용해 HTML/CSS를 즉각적으로 이미지로 렌더링해 주는 이 기능을 활용하면 퀄리티 높고 가벼운 동적 이미지를 생성하기 매우 좋습니다.
그리고 네임카드의 전체적인 톤앤매너(디자인 분위기)를 붓글씨와 여백이 강조된 '전통적인 한국(Traditional)' 스타일과, 아니면 K-Pop 앨범 커버처럼 세련되고 팝한 '현대적인 한국(Modern)' 스타일 2가지 형태 중 선택하면 좋을 것 같습니다.




사용자가 자신의 취향에 따라 '전통적인 스타일(Traditional)'과 '현대적인 스타일(Modern)' 중 하나를 선택하여 네임카드를 생성하게 하는 것은, 서비스의 재미 요소와 고객 만족도를 동시에 잡는 매우 영리한 전략입니다.
이 두 가지 스타일을 Naming-Link의 수익 모델과 결합하여 구현하는 가장 효율적인 아키텍처를 제안합니다.
UX 설계: '스타일 선택' 게이트웨이
결과 화면에서 공유 버튼을 누르기 전, 사용자가 선택할 수 있는 [스타일 선택 모달(Modal)]을 띄웁니다.
선택 옵션:
Style A (Traditional): 한지 배경, 붓글씨 폰트, 낙관 도장 강조.
Style B (Modern): 깔끔한 그라데이션 배경, 고딕/산세리프 폰트, 세련된 레이아웃.
제한 전략:
기본 제공: 무료 사용자는 2가지 스타일 중 1개만 선택하여 생성할 수 있게 합니다.
업셀링(Upselling): 나머지 1개의 스타일도 갖고 싶거나, 워터마크를 제거하고 고화질로 저장하고 싶다면 '디지털 프리미엄 패스($1.99 결제)'로 유도합니다.
'꼭 결제가 아니더라도': 대안적인 리워드 설계
결제 장벽을 느끼는 유저들을 위해, 돈 대신 '바이럴(노동)'을 지불하게 하는 구조를 추가하세요.
"친구에게 링크 공유하고 다른 스타일 무료로 열기"
사용자가 자신의 고유 링크를 친구에게 공유하고, 그 친구가 웹사이트에 접속하면 자동으로 잠겨 있던 스타일이 해제됩니다.
대표님 입장에서는 신규 유저 1명을 0원의 비용으로 확보하게 되므로, $1.99 결제보다 더 큰 이득일 수 있습니다.
"인스타그램 스토리 언급(Tag) 인증"
인스타그램에 결과물을 공유하고, naminglink 계정을 태그하거나 특정 해시태그를 달았을 때, 시스템이 이를 감지(또는 사후 인증)하여 스타일을 해제해 주는 방식입니다. (자동화가 어렵다면, 결제 유도가 더 효율적일 수 있습니다.)
구현 기술 (Next.js & Vercel)
이미지 템플릿 관리: Traditional과 Modern 두 가지 HTML 템플릿을 코드 내에 별도 컴포넌트로 분리합니다.
스타일 토글: 유저가 모달에서 선택한 값을 상태값(useState 등)으로 관리하여, html2canvas나 @vercel/og가 해당 컴포넌트를 렌더링하게 합니다.
조건부 로직:
JavaScript
if (isPaid || isShared) {
    // 스타일 A, B 모두 허용
} else {
    // 선택한 1개만 허용
}
이렇게 하면 결제를 원하는 유저는 바로 $1.99를 지불하고, 결제를 꺼리는 유저는 자연스럽게 친구에게 공유하게 되어 서비스가 자동으로 확장되는 '선순환 바이럴 루프'가 완성됩니다.
기획대로 '전통'과 '현대' 스타일을 동시에 보여주면 유저들이 훨씬 더 개인화된 경험을 한다고 느낄 것입니다. 붓글씨 스타일을 구현할 때 사용할 만한 아름다운 한글 무료 폰트(구글 폰트 등) 확보가 필요해 보입니다.
