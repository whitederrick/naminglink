## 0. 선행 정정

어제 요약에서 계정 삭제 기능 부재를 "방침에 삭제를 약속했는데 미구현"으로 분류했으나, 약관 원문을
확인한 결과 **정정이 필요하다**.

- 개인정보처리방침(`src/lib/legal-content/ko.ts:118`): "이용자는 개인정보 열람, 정정, 삭제, 처리정지,
  동의 철회를 요청할 수 있습니다. **요청은 고객센터 이메일로 접수하며, 본인 확인 후 처리합니다.**"
- 이용약관에 회원 탈퇴·해지 조항 **없음**(`탈퇴|해지|계정 삭제` grep 0건).

즉 **자가 삭제 UI는 약속된 바 없다.** 이메일로 들어온 요청을 이행할 수단(관리자 액션)이 없는
**운영 공백**이지 문서 위반이 아니다. 우선순위는 아래 A보다 낮다.

반면 **배송 상세정보 파기는 명시적 약속인데 메커니즘이 전혀 없다** — 2단계의 실제 위반 항목은 A뿐이다.

## A. 도장 주문 PII 파기 [실제 위반 · 최우선]

### A-1. 약속한 내용

- 방침 "3. 보관 및 파기"(`ko.ts:96`): "배송 상세정보는 주문 처리와 반품·분쟁 대응에 필요한 기간이
  지나면 **파기하거나 비식별화합니다.**"
- 같은 절(`ko.ts:95`): "결제·주문 거래기록은 관계 법령의 법정 보관 기간에 따라 **별도로 보관**합니다."
- §5 국외이전(`ko.ts:110`): Supabase로 "굿즈 주문 시 **주문자명·연락처·배송지**" 이전을 고지함.

→ 파기 의무와 거래기록 보존 의무가 함께 있으므로, **레코드 삭제가 아니라 PII 필드만 선택 제거**해야 한다.

### A-2. 실제로 저장되는 항목

`src/app/api/goods/stamp-order/route.ts:111-132`

| 위치 | 필드 | 성격 |
| --- | --- | --- |
| `orders.customer_name` | 수령인 이름 | 파기 대상 |
| `orders.customer_email` | 주문자 이메일 | 파기 대상 |
| `orders.shipping_address` | 배송지(글로벌은 `[국가] 주소`) | 파기 대상 |
| `orders.metadata.phone` | 연락처 | 파기 대상 |
| `orders.metadata.country` | 배송 국가 | 파기 대상 |
| `orders.metadata.note` | 요청사항(자유 입력 500자) | 파기 대상 |
| `orders.metadata.stampName` | 도장에 새길 문구 | **결정 필요**(이름 자체) |
| `orders.metadata.stampModel` `productCode` `provider` | 상품 사양 | 유지 |
| `orders.metadata.transactionId` `paidAt` | 결제 기록(`stamp-confirm/route.ts:71` 병합) | 유지 |
| `orders.payment_*` `fulfillment_status` `created_at` | 거래기록 | 유지 |

`metadata`는 결제 확인 단계에서 기존 값에 병합되므로, **키 단위 선택 삭제**가 필요하다(통째로 비우면
거래기록인 `transactionId`·`paidAt`까지 날아간다).

### A-3. 현재 파기 메커니즘 — 도장 주문은 아예 대상이 아님

`src/app/api/cron/premium-cleanup/route.ts:182-198`의 주문 PII 정리는 **프리미엄 세션에 연결된
UNPAID 주문의 `customer_name`·`customer_email`만** 지운다. 대상 선정이 `premium_analysis_sessions`
조회에서 출발하는데 도장 주문에는 세션이 없어 **조회에 한 번도 걸리지 않는다.**

결과적으로 다음 두 경우 모두 PII가 무기한 잔존한다.

1. **미결제 이탈 주문** — 폼을 채우고 결제하지 않은 주문. 이름·연락처·주소가 그대로 남는다.
   **보관 근거가 아예 없어 가장 문제.**
2. **결제·배송 완료 주문** — 방침이 파기를 약속한 대상인데 기한이 지나도 남는다.

`shipping_address`, `metadata.phone`, `metadata.note`는 **어떤 경로로도 삭제되지 않는다.**

### A-4. 필요한 작업

1. **UNPAID 이탈 주문 파기** — 생성 후 N시간 경과 시 PII 제거. 프리미엄과 같은 기준
   (`PREMIUM_ABANDONED_TTL_HOURS`, 기본 24h) 재사용 권장.
2. **PAID 주문 파기** — `fulfillment_status = COMPLETED` 이후 **90일** 경과 시 배송 PII만 제거,
   거래기록은 유지. 기준 시각은 `updated_at`(COMPLETED 전환 시점)이며, 배송완료 시각을 별도
   컬럼으로 두지 않아도 되는지 구현 시 확인한다.
3. **크론 배치** — `premium-cleanup`에 단계를 추가하는 방식 권장. GitHub Actions 시간별 스케줄과
   `CRON_SECRET`을 그대로 재사용하므로 신규 워크플로·시크릿이 필요 없다.
4. 관리자 주문 표에 파기 예정일 노출(선택).

마이그레이션은 불필요하다(기존 컬럼·JSONB 갱신만).

### A-5. 확정된 정책 (사용자 결정, 2026-07-24)

- **배송 PII 보관 기간: 배송완료(`fulfillment_status = COMPLETED`) 후 90일.**
  해외 배송(US$34.99 도장) 분쟁이 늦게 들어올 가능성을 고려한 선택.
- **`stampName`: 배송 PII와 같은 시점에 함께 파기.**

#### `stampName`이 본명이 아니어도 개인정보인가 (검토 결과)

**같은 행에 수령인·연락처·주소가 살아 있는 동안에는 본명 여부와 무관하게 개인정보다.** 개인정보보호법
제2조 1호는 그 자체로 개인을 알아볼 수 있는 정보뿐 아니라 **다른 정보와 쉽게 결합하여 알아볼 수 있는
정보**까지 포함하며, 도장 문구는 배송지·연락처와 한 레코드에 있다.

90일 뒤 배송 PII를 지운 다음에도 다음 두 가지가 남는다.

1. **`orders.user_id`는 파기 대상이 아니다**(계정 페이지 주문 내역에 필요). 회원 주문이면 문구는
   여전히 특정 회원과 연결된 상태로 남는다. 비회원 주문은 `user_id`가 null이라 사실상 비식별이다.
2. **이 상품은 문구가 본명인 경우가 다수일 것으로 봐야 한다.** 주문 폼의 필드가 "새길 이름"이고
   검증이 한글·한자 1~8자(`stamp-order/route.ts:29-32`)로 인명 형태다. 저장 시점에 본명인지
   아호·상호인지 구분할 방법이 없으므로 안전한 쪽을 기본값으로 삼는다.

반대 논거도 있다. 도장 문구는 "공급한 재화의 내용"이라 전자상거래법상 대금결제·재화공급 기록의
일부로 볼 여지가 있다. 다만 **무엇을 공급했는지는 `stampModel`·결제금액·주문일로 충분히 특정되고**,
문구 원문이 꼭 필요한 상황은 동일 도장 재제작 정도인데 90일이 지났다면 신규 주문으로 받는 것이
자연스럽다. 따라서 함께 파기로 정한다.

## B. 계정 삭제 이행 수단 [운영 공백 · A 다음]

### B-1. DB는 이미 삭제에 맞게 설계돼 있다

| 대상 | 외래키 동작 | 삭제 시 결과 |
| --- | --- | --- |
| `naming_logs.user_id` | `on delete cascade` | 저장한 작명 결과 자동 삭제 |
| `orders.user_id` | `on delete set null` | **거래기록 보존, 회원 링크만 끊김** |
| `premium_analysis_sessions.user_id` | `on delete set null` | 24h 크론으로 어차피 파기 |
| `ai_usage_logs` · `ad_events` | `user_id` 컬럼 자체가 제거됨(`20260715150000`) | 연결 없음 |

`orders`가 `set null`인 것이 핵심이다. 전자상거래법상 계약·청약철회·대금결제 기록 보관 의무와
개인정보 파기 요구를 동시에 만족하는 형태다. **추가 마이그레이션 없이 auth 사용자만 지우면 된다.**

### B-2. 필요한 작업 (작음)

- `/api/admin/operations`에 `DELETE_USER` 액션 추가. 현재 `DISABLE_USER`/`ENABLE_USER`가
  `route.ts:96-100`에 있고 같은 자리에 들어간다.
- `supabase.auth.admin.deleteUser(userId)` 호출 + 자기 자신 삭제 방지(기존 DISABLE 가드와 동일 패턴).
- `AdminOperationsConsole`에 삭제 버튼과 확인 단계(되돌릴 수 없음).
- 처리 이력 기록 권장. 단 **이력에 이메일을 남기면 그 자체가 PII 잔존**이므로 `user_id`만 남긴다.
- 관리자 화면은 한국어 전용이라 **i18n 비용 0**.

### B-3. 자가 탈퇴 UI는 지금은 권장하지 않음

`src/lib/i18n-account.ts`는 **23개 로케일 사전**이라 문구 추가 비용이 크고, 방침이 이미 이메일 접수로
규정하고 있어 의무 이행에 불필요하다. 오픈 후 실제 수요를 보고 결정한다.

## 구현 결과 (2026-07-24)

**A-1·A-2 완료.** `src/lib/goods-pii-cleanup.ts` 신설, `premium-cleanup` 크론에서 세션 정리보다
앞서 호출한다(세션이 0건일 때 조기 반환에 걸려 건너뛰지 않도록). 파기 대상 선정은 `customer_name`이
null이 아닌 것을 "아직 파기 안 함" 표시로 써서 마이그레이션 없이 재처리를 피한다.

`scripts/verify-goods-pii-cleanup.ts`로 공유 Supabase에 실제 주문 6건을 넣고 검증했다(테스트 행은
실행 후 id로 삭제, 잔여 0건 확인).

| 케이스 | 기대 | 결과 |
| --- | --- | --- |
| 미결제 이탈(48시간 경과) | 파기 | PASS |
| 미결제 신규(1시간) | 보존 | PASS |
| 배송완료 100일 경과 | 파기 | PASS |
| 배송완료 10일 경과 | 보존 | PASS |
| 결제됐지만 배송 미완료(100일) | 보존 | PASS |
| 환불·취소 100일 경과 | 파기 | PASS |

거래기록(`productCode`·`stampModel`·`transactionId`·`paidAt`·`provider`)이 파기 후에도 남는 것을
함께 확인했다.

**범위 추가**: 환불·취소(`fulfillment_status = CANCELLED`)된 유료 주문도 배송지가 그대로 남는
같은 결함이라 A-2에 포함했다. 원래 범위에는 COMPLETED만 있었다.

**B 완료.** `/api/admin/operations`에 `DELETE_USER` 액션(자기 자신 삭제 차단, 처리 이력은 `user_id`만
로깅), 회원 표에 삭제 버튼과 확인 창(무엇이 지워지고 무엇이 남는지 명시). 안내 문구도 갱신했다.
지속적인 감사 테이블은 만들지 않았다 — 필요하면 후속으로 마이그레이션이 필요하다.

**약관 반영 완료.** 개인정보처리방침 "3. 보관 및 파기"에 90일·24시간을 명시하고 23개 로케일에
반영했다. 4개 문서를 통째로 재번역하는 `translate-legal-content.mjs` 대신 해당 문단만 교체하는
`scripts/translate-legal-paragraph.mjs`를 새로 만들어 썼다(기존 번역 churn 방지, 파일당 1줄 변경).

**미결**: `LEGAL_EFFECTIVE_DATE`(현재 2026-07-22)는 그대로 두었다. 4개 문서가 공유하는 상수라
바꾸면 변경되지 않은 약관·환불·요금까지 새로 시행되는 것으로 표시된다. 법무 검토 반영 등 추가
수정이 더 남아 있으므로 **런칭 직전에 한 번만 갱신**하는 것을 권한다.

## 진행 순서

1. **A-1 미결제 이탈 주문 파기** — 보관 근거가 전혀 없는 데이터이고, 실주문이 들어오기 전에 켜두면
   소급 정리가 필요 없다.
2. **A-2 결제 완료 주문 배송 PII 파기** — 보관 기간 결정 후 착수.
3. **B 관리자 삭제 액션** — 삭제 요청이 실제로 들어오기 전에만 있으면 된다.

착수 전 결정 2건은 2026-07-24에 확정됐다(A-5 참조). 남은 미결은 방침 문구 갱신 여부다 —
현재 "필요한 기간이 지나면"으로만 적혀 있어 90일을 명시할지 결정이 필요하다. 명시하면 이용자에게
더 명확하지만 기간 변경 시 23로케일 약관을 다시 손대야 한다.
